Rewarding the Rare: Uniqueness-Aware RL for Creative Problem Solving in LLMs

A method to prevent exploration collapse in RL-trained LLMs by rewarding diverse solution strategies

RL training often causes LLMs to converge to a single dominant reasoning pattern - this paper calls it exploration collapse. The model gets better at its favorite approach (improving pass@1) but loses the ability to try different approaches (hurting pass@k).

The solution is elegantly simple: use an LLM judge to cluster rollouts by high-level strategy, then upweight advantages for rare but correct strategies. Common correct strategies still get positive rewards, just less than unique ones. This "rewards the rare" approach maintains diversity without sacrificing accuracy.

The Problem: Exploration Collapse

Core Problem
Token-level diversity measures are a poor proxy for strategy-level diversity. Two solutions can differ in wording but share the same mathematical approach.

Consider solving x2 - 5x + 6 = 0. Two solutions might both use the quadratic formula but present it differently - one showing intermediate steps, the other simplifying immediately. Token-level metrics would call these "diverse," but they're actually the same strategy. Meanwhile, factoring to (x-2)(x-3) = 0 is genuinely different.

Key Insight

Pass@k performance depends on having multiple conceptually distinct strategies across k samples, not merely producing superficial token-level variation.

Formally, token-level entropy Ht = -sum p(v|x<t) log p(v|x<t) averaged over positions does NOT imply high strategy diversity. Prior methods like DAPO (entropy bonuses) and Forking Token (low-probability regularization) operate at this token level and cannot distinguish "same approach, different wording" from "genuinely different approach."

Methodology

Method in One Sentence
Cluster rollouts by strategy using an LLM judge, then multiply GRPO advantages by 1/cluster_size - rare correct strategies get boosted, common ones get dampened.

Key Takeaway
Correctness alone doesn't determine advantage - among correct solutions, rare strategies get rewarded MORE than common ones, preventing collapse to a single approach.

Strategy Clustering

For each training problem, the policy generates K=8 rollouts. An LLM judge (larger model from same family, e.g., 32B for 7B policy) examines all K rollouts and partitions them into clusters based on high-level solution strategy, ignoring surface differences like variable naming.

Uniqueness Weight

Each rollout gets a uniqueness weight w = 1/(cluster_size)alpha. Singletons (rare strategies) get w=1; rollouts in clusters of 4 get w=1/4alpha. The alpha parameter controls how much to favor rarity.

Combined Advantage

Final advantage = uniqueness_weight ร GRPO_advantage. This multiplicative combination ensures wrong answers never become attractive (negative ร positive = still negative), while differentiating among correct answers by rarity.

Technical Details

\[ w_{m,k} = \frac{1}{f_{m,k}^\alpha} \]
Equation 4: Uniqueness Weight

fm,k = cluster size (number of rollouts sharing same strategy)

alpha = hyperparameter in [0,1] controlling reweighting strength

With alpha=0, all weights are 1 (standard GRPO). With alpha=1, weight equals exactly 1/cluster_size.

\[ \text{advantage}_{m,k} = w_{m,k} \cdot z_{m,k} = \frac{1}{f_{m,k}^\alpha} \cdot \frac{r_{m,k} - \mu_m}{\sigma_m + \varepsilon} \]
Equation 5: Uniqueness-Weighted Advantage

The GRPO term zm,k handles correctness (positive for correct, negative for incorrect). The uniqueness weight wm,k handles rarity. Their product gives the final advantage.

Implementation Details

Pseudocode 
def compute_advantage(reward, mean_reward, std_reward, cluster_size, alpha, eps=1e-8):
    # Standard GRPO group-normalized advantage
    z = (reward - mean_reward) / (std_reward + eps)
    # Uniqueness weight
    w = 1.0 / (cluster_size ** alpha)
    # Combined advantage
    return w * z
Hyperparameters

Learning rate: 5e-7 (AdamW)
Rollouts per problem: K=8
Generation temperature: 1.0
KL coefficient: 0.001
Max tokens: 4096 (Qwen2.5), 20480 (Qwen3/OLMo)

Practical Tips ▼
  • Use a judge model significantly larger than the policy (32B for 7B, 72B for 7B)
  • Cache strategy clusters if memory allows
  • Monitor entropy curves - should stay stable, not decrease
  • Log cluster size distribution to verify meaningful partitions
  • Track both pass@1 (should not decrease) and pass@K (should increase)

Key Insights

Bottom Line
If you want your LLM to find DIFFERENT correct solutions (not just one solution many times), this method provides a principled way to maintain exploration during RL training.
Insight 1: Token Diversity โ Strategy Diversity

Two solutions can have completely different tokens while implementing the exact same approach. Token-level measures like entropy see these as "diverse," but from a problem-solving perspective, they're redundant.

Insight 2: Multiplicative Combination is Key

Multiplying uniqueness weight by GRPO advantage ensures wrong answers never become attractive (negative ร positive = still negative). You can't just ADD a novelty bonus.

Insight 3: Inference-Only Judge

The LLM judge runs in inference-only mode (no training cost). Can leverage any capable model as judge, and quality improves with model scale.

Quotable Takeaways

  • "Reward the rare" - boost correct solutions that use uncommon strategies
  • "Token diversity is not strategy diversity" - fundamental insight motivating the work
  • "Exploration collapse: high pass@1, flat pass@k" - problem definition
  • "Let an LLM judge cluster by strategy, then upweight small clusters" - method summary

Limitations and Open Questions

Acknowledged Limitations

  • LLM judge introduces computational overhead
  • Judge may miscluster on ambiguous problems
  • Definition of "high-level strategy" is task-dependent
  • Method measures within-problem diversity, not cross-problem novelty

Open Questions

  • What is the optimal alpha for different domains?
  • Can the method work with smaller judges?
  • Could this combine with other exploration methods?
  • Does it help on non-reasoning tasks (creative writing, code)?

When to Use This Method

Good Fit

Reasoning tasks with multiple valid approaches, when pass@k matters (best-of-N inference), when you have access to a larger judge model, when you observe pass@1 improving but pass@k stagnating.

Not Recommended

Single-answer tasks with no strategy variation, when pass@1 is the only metric, when judge inference cost is prohibitive, very short horizon tasks.
