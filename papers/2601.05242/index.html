GDPO: Group reward-Decoupled Normalization Policy Optimization

A fix for multi-reward RL that preserves training signal granularity

Authors: Shih-Yang Liu, Xin Dong, Ximing Lu, et al. (NVIDIA)
Code: HF-TRL, verl, Nemo-RL implementations available

When training LLMs with multiple reward signals (accuracy + format + length), GRPO sums all rewards then normalizes. This causes advantage collapse: different reward combinations like (0,1) and (0,2) map to identical advantages. GDPO fixes this by normalizing each reward dimension independently before summing.

The Problem: GRPO Reward Collapse

Core Issue
GRPO's sum-then-normalize approach collapses 6 distinct reward combinations into only 2 advantage groups, losing critical training signal.
Figure 2: GRPO vs GDPO Advantage Computation
What This Shows

Each box represents a (reward1, reward2) combination for a rollout. GRPO (left) maps multiple boxes to identical advantages (same color = same advantage). GDPO (right) preserves more distinctions with separate normalization per reward.

Key observation: Under GRPO, (0,1), (0,2), and (1,2) all get the same advantage (-0.7071, 0.7071). Under GDPO, (0,2) correctly gets a larger advantage than (0,1) because it achieved more rewards.

Mathematical Analysis

Consider 2 rollouts with 2 binary rewards each. GRPO computes:

\[ A_{sum}^{(i,j)} = \frac{r_{sum}^{(i,j)} - \text{mean}}{\text{std}} \]
GRPO: Normalize the summed reward

When rollout sums are (0,1), (0,2), or (1,2), the normalized advantages are always (โ0.7071, 0.7071) regardless of which specific rewards contributed. The distinction between achieving 1 vs 2 rewards is lost.

GDPO: The Solution

Method in One Sentence
Normalize each reward dimension separately, then sum the normalized advantages.
Figure 1: GDPO Method Overview and Reward Trends
What This Shows

(a) GDPO's pipeline: per-reward group normalization โ sum โ batch-wise normalization

(b) Training curves: GDPO (green) achieves higher format AND correctness rewards than GRPO (blue), with more stable convergence.

How GDPO Works

  1. Per-reward normalization: For each reward k, compute A_k = (r_k - mean_k) / std_k
  2. Sum normalized advantages: A_sum = A_1 + A_2 + ... + A_n
  3. Batch-wise normalization: Scale to keep magnitudes stable as rewards increase
\[ A_k^{(i,j)} = \frac{r_k^{(i,j)} - \text{mean}\{r_k^{(i,1)}, ..., r_k^{(i,G)}\}}{\text{std}\{r_k^{(i,1)}, ..., r_k^{(i,G)}\}} \]
Equation 4: Per-reward normalization
\[ A_{sum}^{(i,j)} = A_1^{(i,j)} + \cdots + A_n^{(i,j)} \]
Equation 5: Sum normalized advantages

Implementation

GDPO Pseudocode
# GDPO: Normalize each reward separately, then sum
for k in range(n_rewards):
    A[k] = (r[k] - mean(r[k])) / (std(r[k]) + eps)
    
A_sum = sum(A[k] for k in range(n_rewards))

# Batch-wise normalization for stability
A_final = (A_sum - batch_mean) / (batch_std + eps)
Implementation Note

GDPO is a drop-in replacement for GRPO's advantage computation. It's compatible with HF-TRL, verl, and Nemo-RL frameworks.

Why GDPO Helps: Advantage Granularity

Figure 3 shows GDPO preserves 2-4x more distinct advantage groups as rollouts or rewards scale. This gives the policy optimizer more fine-grained signal about which responses are better.

Key Finding

GRPO without std normalization (Dr.GRPO variant) only slightly improves granularity. GDPO's decoupled approach is fundamentally better.

Key Takeaways

When to Use GDPO

Use it: Any multi-reward RL (2+ reward signals), when observing training instability, when rewards have different scales.

Skip it: Single-reward optimization (GDPO = GRPO in this case).
